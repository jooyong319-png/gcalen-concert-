## 2단 위키 구조 (프로젝트 위키 + 통합 위키)
- **이 위키(`whenstage/wiki/`)** — WhenStage 전용. 저장소 안에 있어 원격/클라우드 자동화도
  `git clone`으로 읽는다.
- **통합 위키(`d:/Gcalen/wiki/`)** — Next.js·Vercel·Supabase·SEO 등 **어느 프로젝트든 재사용되는
  기술 지식.** 로컬에서 작업할 땐 그쪽을 먼저 참고하면 검증된 함정·패턴을 알고 출발한다.
  ⚠️ 통합 위키는 이 저장소 **바깥의 로컬 폴더**라 `git clone`만 하는 원격 세션은 못 읽는다.
- **경계 규칙**: "프로젝트 이름을 빼도 문장이 성립하면 → 통합 위키, 프로젝트 이름이 꼭 있어야
  하면 → 이 위키." 예) "Vercel `NEXT_PUBLIC_*`은 Sensitive 두면 빈 값" = 통합. "쿠폰을 회사
  마찰로 뺐다" = 프로젝트(여기).

## 유지 워크플로우 (LLM이 수행하는 3동작)
- **Ingest(갱신)**: 프로젝트에 변화(새 기능/구조 변경/결정)가 생기면 → 관련 페이지 갱신 +
  [[decisions]](결정이면) 추가. 변경 로그는 저장소 루트 `PROJECT_STATUS.md`의 "최근 변경 로그"가
  담당(이 위키의 log 역할). 재사용 가능한 **기술** 지식이면 여기가 아니라 통합 위키로 보낸다.
- **Query(조회)**: 질문 받으면 → 관련 페이지 근거로 답 + 페이지명 인용. 코드가 진실의 원천이니
  파일/함수/플래그를 인용할 땐 실제로 있는지 확인 후 신뢰.
- **Lint(점검)**: 가끔 → 문서와 코드가 어긋난 곳, 모순, 오래된 서술, 고아 페이지 점검·수정.
  문서와 코드가 다르면 **코드가 맞다**(문서가 stale).
